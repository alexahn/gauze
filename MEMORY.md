# Gauze repo notes

- Entity-style GraphQL method coverage lives under `test/<type>/(database|system)/methods/` and uses numbered step files like `0.create.js`, `1.count.js`, loaded with `load_steps(...)`.
- Database method suites call `run_steps(...)` with the database GraphQL schema; system method suites either also use `run_steps(...)` directly or, for access-controlled entities like `ytitne`, `run_layers(...)` with `./environment` steps first.
- `count` tests in this repo follow the same two-step pattern as other method tests: create a single record, then assert the GraphQL `count_<type>` query returns `[{"select":"count(*)","count":1}]`.
- In the whitelist/blacklist and relationship system policy matrices, `count` is not a straight copy of `read`: several combinations have different authorization outcomes, and whitelist/blacklist failures use line `4` in the GraphQL error location because the generated count query starts on the opening backtick line.
- Do not assume `count` should share `read` authorization semantics just because both are query methods. In Gauze, whitelist/blacklist policies are method-specific, and when reviewing failures I need to confirm the intended contract from the user before treating read/count mismatches as implementation bugs.
- Gauze is best understood as a realm-based GraphQL framework, not a single flat API: `environment` is the outer auth/session boundary, `system` is the higher-level application API, and `database` is the lower-level CRUD/data realm, all served from one Koa app.
- The core pattern is metadata-driven entity definition. Files in `src/abstract/entities/` define fields, serializers, allowed agent types, method privacy, and relationship-ready metadata; the kernel linker and per-realm GraphQL schemas turn that into concrete GraphQL operations and types.
- The `system` realm typically proxies work into the `database` realm via generated GraphQL operations rather than talking to storage directly, so cross-realm delegation is a central architectural idea.
- Authentication is session-backed JWT handoff across realms. Agents authenticate in `environment`, then receive realm-specific JWTs for `system` or `database`; request handling verifies both the token and the backing session record before executing GraphQL and committing/rolling back transactions.
- The bundled `src/views/project` frontend is mainly an operational/admin console for browsing headers/entities and managing environment/proxy/system sessions, not just a generic marketing or consumer-facing UI.
- `src/database|system/.../ezuag.*` and `src/database|system/.../ytitne.*` are the source templates for the shared base generation flow. To propagate edits from those templates into the rest of the generated code, run `./bin/base_from_src` first and then `./bin/src_from_base`; running `src_from_base` alone will overwrite direct edits to generated files from the existing base.
- When changing core query semantics like `order`, the correct implementation boundary is the shared kernel runtime in `src/kernel/src/models/database.js`, not entity-specific template models like `src/database/models/ezuag.js` or `src/database/models/ytitne.js`. Template edits only make sense for generation shape, not shared database behavior.
- After regenerating from `base`, frontend GraphQL callers in `src/views` should be updated generically, not with per-entity checks for template names like `ezuag` or `ytitne`. If the backend schema changed everywhere, the view services should assume the new schema everywhere too.
- Query-time `order_direction` and `order_nulls` were removed from the runtime and GraphQL query args; only entity-level `default_order_direction` remains, and it still drives fallback sort direction when an order item omits `order`.
- `where_between` now supports composite cursor-style filtering in the shared runtime at `src/kernel/src/models/database.js`: when a non-primary field range is provided alongside `where_between[primary_key]`, Gauze treats the primary key as a tie-breaker and emits tuple-style exclusive bounds like `(field > start) OR (field = start AND pk > start_pk)` and `(field < end) OR (field = end AND pk < end_pk)`. Plain primary-key-only `where_between` still uses normal `whereBetween`.
- Open-ended `where_between` bounds use `null` as the open side. Plain ranges are inclusive (`[start, null]` -> `>= start`, `[null, end]` -> `<= end`), while composite non-primary + primary-key ranges keep cursor-style exclusive bounds on the provided side. Primary-key shard routing also treats null range sides as spanning to the table shard minimum/maximum.
- Plain GraphQL read/count operations still expose raw `where_between`, `limit`, and `order`; the first-class cursor/page-info API exists only in the `ytitne`/`ezuag` pilot. Regular user-supplied composite `where_between` remains a value-range feature, while cursor pagination uses an internal direction-aware lexicographic boundary.
- The cursor pagination pilot is intentionally exposed only for `ytitne` and `ezuag` as `cursor_read_*`, `cursor_update_*`, and `cursor_delete_*`; there is no `cursor_create_*`. Cursor tokens are signed base64url JSON request tokens that embed the original operation parameters plus previous/current/next page filters, and cursor continuation calls must not mix `cursor` with other client operation arguments.
- Cursor continuation must allow system-injected `cache_where_in` and `cache_where_not_in` because those filters enforce element-level access control. Do not embed those cache filters in outgoing cursor payloads; regenerate/merge them per request from the system authorization layer.
- Empty cursor pages should preserve navigation from the decoded cursor payload. For example, a `previous_cursor` before the first row may legitimately return zero nodes, but its `page_info.next_cursor` should point back to the embedded current page filter so the client can move forward again.
- Cursor fields should align with availability flags: when `page_info.has_previous_page` is false, `previous_cursor` should be null; when `page_info.has_next_page` is false, `next_cursor` should be null. Preserve empty-page recovery for stale/out-of-window cursor tokens through the decoded payload, not by exposing unavailable page cursors.
- Cursor GraphQL page-info types live in the structure realm at `src/structure/gauze/cursor.js`; database/system cursor query and mutation interfaces should reference `$structure.gauze.cursor.*` rather than defining local `interfaces/graphql/cursor.js` files.
- Cursor pagination must not restrict GraphQL `order` parameters: it should honor ascending, descending, mixed multi-column order, explicit `nulls`, and the primary-key tie-breaker. Cursor filters are internal lexicographic boundaries, not simple primary-key ranges; do not route shards from a cursor primary-key-looking bound unless the cursor order itself proves that range is valid.
- Database connection routing should use `where_between[primary_key]` as a positive shard constraint: route only to current shards whose inclusive `[start, end]` range overlaps the requested primary-key range, including bootstrapped `where: {}` and combined non-primary `where_in` cases.
- Database connection routing should treat shard-producing filters on the same table primary key as cumulative constraints: exact `where[primary_key]`, `where_in[primary_key]`, and `where_between[primary_key]` are intersected with the current candidate shard set. For relationship traversals, relationship-derived candidate shards are also intersected with those primary-key filters. For relationship/access table reads, from/to/entity/agent anchors keep anchor-shard routing; only the unanchored fallback path applies row primary-key shard filters on the relationship/access table shard domain.
- Sharding connection-routing tests live in `test/sharding/routing.js`; do not recreate the older `test/sharding/where_between.js` file when adding routing coverage.
- Sharded database method aggregation must sort flattened row results after `Promise.all(...)`; per-shard SQL ordering is not enough. Use the shared database model row-sort helper so root/relationship read, update, and delete results share ordering semantics and primary-key tie-breaking.
- Root and relationship database update/delete targeting goes through the corresponding read transaction first and then mutates only `whereIn(primary_key, valid_ids)`. Do not reapply or intersect the original filters in the mutation SQL; the selector read already handled them. Update methods reread by primary key after mutation so they return current row state; relationship update rereads must preserve `parameters.source` because system-layer delegation can pass relationship context through mutation arguments instead of resolver scope. Delete methods return the pre-delete selector rows. `_chunk_action` must pass its chunked parameter copy to the action, not the original unchunked parameters.
- Count GraphQL responses use the custom `CountValue` scalar instead of GraphQL `Int` because PostgreSQL returns `count(*)` as an int8 string. Count merge paths normalize count values with `BigInt`, preserve JSON numbers for safe integer-only SQLite-style results, and return strings for PostgreSQL/string or unsafe totals.
- Count result construction uses the requested `count` input map as the source of truth for response `select` labels, falling back to a canonical default `count(*)` selector when `count` is omitted. Avoid synthetic empty-count selector hashes because they do not model the client's requested result shape.
- The project UI graph route is `project.system.headers.graph` at `/project/x/headers/graph`. It is project-local React state in `src/views/project/src/components/Graph.jsx`, uses `gauzemodel` for generic header reads/counts, and does not reuse the older `src/views/gauze/v1` graph service or root-agent bootstrap.
- Project graph relationship traversal should use the same GraphQL `source` shape as the old graph: `{ _metadata: { type: <source graphql_meta_type>, id: <source id> }, _direction: "from"|"to" }`, then read/count the target header with that `source`.
- Do not add depth-based table color cycling to the project graph unless explicitly requested again; the current requested state is the static pre-color-scheme styling.
- Project graph traversal edges should anchor to visible row centers by row id, not to the table or node midpoint. A `"to"` traversal draws from the parent source row to each child result row; a `"from"` traversal draws from each child result row back to the parent source row.
- Project graph relationship table headers should display the relationship anchor side for the source id: a backend traversal `_direction: "to"` means the opened target table is anchored `from: <source id>`, while `_direction: "from"` should display `to: <source id>`.
- Project graph row edit buttons should open an in-graph single-row item table in `update` mode, not navigate to the item route. The item table has `read`, `update`, and `delete` modes, and update/delete actions must require the same text prompt confirmations as `Item.jsx`.
- Project graph table Create buttons should open an in-graph item table in `create` mode, prepopulate create attributes from the source table's `where` filter or, when no exact filter exists, `where_like`, pass through the source table's relationship `source` object when present so the created item is related back to that table, and require a `"create"` prompt before calling create.
- Project graph table nodes should include a new-tab link to `project.system.headers.header.list`, serializing the node's current `variables` into the list route's `variables` search parameter so the applied table view is preserved.
- Project graph table settings live behind the table node header gear button as an in-node settings view that replaces the table view; field visibility is local display state with a scrollable/searchable field selector, and sort changes apply through the normal table reload path by updating `variables.order`.
- Knex `orderBy` with `nulls` on SQLite clients (`sqlite3` and `better-sqlite3`) generates only a nullness expression such as `order by (text is not null) asc`, which is syntactically valid but does not sort by the column value. Backend SQLite ordering should either strip `nulls` or add a secondary column order after the nullness expression.
- Project graph item view nodes should keep `parentNodeID` and `parentEntityID` so the graph can draw a visible edge from the source row to the item view, using the same measured row-anchor system as relationship traversal. Create-mode item nodes start without a parent row id, then adopt the created item id as `parentEntityID` after create succeeds so the edge appears from the newly inserted source-table row to the read-mode item view.
- Project graph create-mode item nodes should still render a table-level `"create"` edge back to the source table before the item exists; after create succeeds, the node switches to the normal row-anchored item edge using the created item id.
- Project graph row whitelist/blacklist buttons should create five in-graph WHITELIST/BLACKLIST table nodes, one per access method (`create`, `read`, `update`, `delete`, `count`), each prefiltered with `gauze__<kind>__entity_id`, `gauze__<kind>__entity_type`, `gauze__<kind>__method`, and realm when the system agent audience is available; do not navigate out of the graph.
- Project graph access buttons should render whitelist with the filled bookmark icon and blacklist with the outline bookmark icon.
- Project graph table scroll containers should stop wheel propagation so scrolling a table does not also trigger graph viewport zoom.
- Project graph title metadata next to the entity name, such as traversal source ids and item ids, should wrap instead of using `truncate`; this is the top header area of the table/node, not the column header row.
- Project graph item nodes should show mode-level GraphQL errors for create/update/delete in the top `project-graph-error` banner, not only near the bottom apply row. Field validation errors still render beside the field input.
- Project graph item field validation errors should render below the related input in normal document flow, not absolutely positioned or inline to the right; this keeps messages readable without horizontal scrolling and prevents the last field's error from covering the apply row.
- Project graph table column names and item field-name cells should use the same full-value popover behavior as data values, while staying visually truncated in the cell.
- Project graph table values and item read-mode values remain visually truncated in cells, but clicking a value should open a graph-specific Popover showing the full value; the popover content must preserve long text/JSON with wrapping and avoid table-scroll clipping.
- Do not render relationship traversal menus as project graph table-cell popovers: popover content can leak/clamp inside the table in some render states. Use a compact in-row selector or another control whose menu body is not regular table content.
- For project graph drag/pan performance, prefer separating graph node layout state (`x`, `y`, viewport transforms) from node content state and memoizing the table/item contents with stable action callbacks. Avoid imperative DOM transform hacks as the primary solution.
- Project UI 401 handling should classify the failed request JWT by decoded `aud` rather than calling `getEnvironmentJWT()`, `getProxyJWT()`, or `getSystemJWT()`: those getters delete expired tokens as a side effect, so they can misclassify an expired system token and send the user through nested signin redirects. When clearing a token after a 401, only delete it if the raw stored token still equals the failed request token so stale in-flight responses cannot wipe a newer session.
- Sharded result merge ordering should compare the JavaScript values returned by Knex's drivers rather than stringifying every value. `better-sqlite3` returns SQLite values as `null`, `number`/`BigInt`, `string`, and `Buffer`; `node-postgres` uses parsers for common PostgreSQL types and returns unknown types as strings. Avoid `String(...)` fallback in comparators because objects such as JSON collapse to `"[object Object]"`.
- PostgreSQL is intended to be a supported database runtime; `pg` is an explicit runtime dependency so Knex can use node-postgres and local code can inspect `pg.types`/`pg-types` behavior when implementing PostgreSQL support.
- PostgreSQL shard-merge ordering coverage lives in `test/sharding/ordering.js` and uses `pg.types.getTypeParser(...)` directly, without a live PostgreSQL server, to exercise the JavaScript values returned for `bytea`, arrays, point/circle, interval, JSONB, and int8/numeric strings.
- Database ordering behavior is documented under `docs/database/ordering.md`; keep it in sync with the shared comparator and sharded merge sorting in `src/kernel/src/models/database.js`.
